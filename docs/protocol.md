# eufy BLE体重計 クリーンルーム実装向けプロトコル仕様

## 目的

このドキュメントは、eufy系BLE体重計からローカルに測定値を取得する
独立実装向けの仕様メモです。外部から観測できるBLEの挙動、GATT構成、
送受信バイト列、パケット形式だけを扱います。

主目的は次の範囲です。

- クライアント起動中に近くの体重計を検出する。
- 体重計が起きたタイミングでBLE接続する。
- 通知を購読する。
- 安定体重を標準出力やログへ出す。

クラウドログイン、サーバ同期、アプリ内DB、ユーザー自動判定、
体組成の算出式は対象外です。

## 対象ファミリー

実機で取得確認済みなのは `eufy T9120` です。他ファミリーは同じBLE
体重計群として調査対象に含めますが、同じ実装に混ぜず、プロファイルを
分けて扱ってください。

| ファミリー                     | 状態                | 実装方針                                                             |
| ------------------------------ | ------------------- | -------------------------------------------------------------------- |
| `T9120` / `T9146` / `T9147` 系 | `T9120`のみ実測済み | まずこの仕様の主対象として扱う。                                     |
| `T9140 V1/V2/V3` 系            | 未実機確認          | UUIDとpacket形式が異なるため別プロファイルにする。                   |
| `T9148` / `T9149` 系           | 未実機確認          | 追加通知チャネルと認証系があるため別プロファイルにする。             |
| `T9150` / `T9130` 系           | 未実機確認          | MTU、暗号/認証チャネル、ユーザー選択が絡むため別プロファイルにする。 |

未知の機種を見つけた場合は、機種名、広告データ、service UUID、
通知の生バイト列を保存し、既存プロファイルへ無理に当てはめないでください。

## BLE検出

クライアント起動中はBLE scanを継続します。体重計は常時見えるとは限らず、
ユーザーが乗ったタイミングで起きて検出可能になる想定です。

検出時は、local nameとadvertised service UUIDの両方を候補として見ます。
ただし、BLE広告にservice UUIDや完全なnameが必ず載るとは限りません。
片方だけで確定扱いにせず、接続後のservice discoveryと通知内容で
プロファイルを確定してください。

| ファミリー                  | local name候補                                            | service UUID候補                       | 確認状態            | 検出時の扱い                                                           |
| --------------------------- | --------------------------------------------------------- | -------------------------------------- | ------------------- | ---------------------------------------------------------------------- |
| `T9120` / `T9146` / `T9147` | `eufy T9120`, `eufy T9146`, `eufy T9146 C1`, `eufy T9147` | `0000fff0-0000-1000-8000-00805f9b34fb` | `T9120`のみ実測済み | `FFF0`だけでは後続系と区別できないため、nameも見る。                   |
| `T9140 V1`                  | `eufy T9140`                                              | `0000ffb0-0000-1000-8000-00805f9b34fb` | 未実機確認          | `T9120`系とは別プロファイルにする。                                    |
| `T9140 V2`                  | `eufy T9140`                                              | `4143f6b0-5300-4900-4700-414943415245` | 未実機確認          | `T9140 V2`候補として扱う。                                             |
| `T9140 V3`                  | `eufy T9140`                                              | `4143f7b0-5300-4900-4700-414943415245` | 未実機確認          | `T9140 V3`候補として扱う。                                             |
| `T9148` / `T9149`           | `eufy T9148`, `eufy T9149`                                | `0000fff0-0000-1000-8000-00805f9b34fb` | 未実機確認          | `FFF0`だけでは区別できないため、nameと接続後characteristicを確認する。 |
| `T9150` / `T9130`           | `eufy T9150`, `eufy T9130`                                | `0000fff0-0000-1000-8000-00805f9b34fb` | 未実機確認          | 複数通知チャネル候補を前提に、`T9120`系とは分ける。                    |

`FFF0` serviceだけが見え、nameが空または不明なdeviceは `FFF0 unknown` として
扱います。この場合、接続してもすぐに `T9120`系parserへ流さず、
service/characteristic列挙と生通知の保存を優先してください。

推奨挙動:

- クライアントが起動している間だけscanする。
- 強い候補を見つけたらscan windowの終了を待たずに接続する。
- name一致、既知service UUID、接続後のcharacteristic構成で候補を絞る。
- 体重らしいmanufacturer payloadだけでは自動接続しない。
- macOSではBLE MACアドレスに依存しない。CoreBluetoothではホスト内の
  UUID風addressとして見える。

実用上は「6秒scanして、未検出なら10秒周期で再scan」程度でよいです。
callbackで候補検出時に即接続する方式の方が、scan完了待ちより安定します。

## GATTプロファイル

### プロファイルA: T9120 / T9146 / T9147候補

`T9120`で測定値取得まで確認済みのプロファイルです。

| 役割                   | UUID                                   |
| ---------------------- | -------------------------------------- |
| service                | `0000fff0-0000-1000-8000-00805f9b34fb` |
| 通知characteristic     | `0000fff4-0000-1000-8000-00805f9b34fb` |
| 書き込みcharacteristic | `0000fff1-0000-1000-8000-00805f9b34fb` |

接続後はwriteより先にnotifyを購読します。確認済み範囲では応答なしwriteで
動きます。

`T9120`のlive weight取得では、認証ハンドシェイクは観測していません。

### プロファイルB: T9140 V1/V2/V3候補

この系統は `T9120`系とはUUIDもパケット形式も違います。

| variant | service                                | notify                                 | write                                  |
| ------- | -------------------------------------- | -------------------------------------- | -------------------------------------- |
| V1      | `0000ffb0-0000-1000-8000-00805f9b34fb` | `0000ffb2-0000-1000-8000-00805f9b34fb` | `0000ffb1-0000-1000-8000-00805f9b34fb` |
| V2      | `4143f6b0-5300-4900-4700-414943415245` | `4143f6b2-5300-4900-4700-414943415245` | `4143f6b1-5300-4900-4700-414943415245` |
| V3      | `4143f7b0-5300-4900-4700-414943415245` | `4143f7b2-5300-4900-4700-414943415245` | `4143f7b1-5300-4900-4700-414943415245` |

パケットは `ac 02 ...` で始まる系統です。`cf` live packetとしてparseしては
いけません。

### プロファイルC: T9148 / T9149候補

主serviceは `FFF0` 系ですが、追加のreal-data通知チャネルが必要です。

| 役割                       | UUID                                   |
| -------------------------- | -------------------------------------- |
| service                    | `0000fff0-0000-1000-8000-00805f9b34fb` |
| 通知characteristic         | `0000fff4-0000-1000-8000-00805f9b34fb` |
| 書き込みcharacteristic     | `0000fff1-0000-1000-8000-00805f9b34fb` |
| real-data通知候補          | `0000fff2-0000-1000-8000-00805f9b34fb` |
| device information service | `0000180a-0000-1000-8000-00805f9b34fb` |
| firmware revision          | `00002a26-0000-1000-8000-00805f9b34fb` |
| battery service            | `0000180f-0000-1000-8000-00805f9b34fb` |
| battery level              | `00002a19-0000-1000-8000-00805f9b34fb` |

この系統は認証や追加チャネルの確立が必要な可能性があります。`T9120`系の
初期化シーケンスだけで測定値が出る前提にしないでください。

### プロファイルD: T9150 / T9130候補

この系統は `FFF0` serviceを使いますが、通知・書き込み・real-dataチャネルの
分担が `T9120`系と異なります。

| 役割                       | UUID                                   |
| -------------------------- | -------------------------------------- |
| service                    | `0000fff0-0000-1000-8000-00805f9b34fb` |
| primary通知候補            | `0000fff1-0000-1000-8000-00805f9b34fb` |
| 書き込み/暗号チャネル候補  | `0000fff2-0000-1000-8000-00805f9b34fb` |
| real-data通知候補          | `0000fff3-0000-1000-8000-00805f9b34fb` |
| device information service | `0000180a-0000-1000-8000-00805f9b34fb` |
| firmware revision          | `00002a26-0000-1000-8000-00805f9b34fb` |
| battery service            | `0000180f-0000-1000-8000-00805f9b34fb` |
| battery level              | `00002a19-0000-1000-8000-00805f9b34fb` |

MTU拡張、認証/暗号系のやり取り、体重計側のユーザー選択が絡む候補です。
`T9120`系と同じプロファイルとして扱わないでください。

## T9120系の接続フロー

`T9120`系プロファイルでは、次の流れを基本にします。

1. クライアント起動中にscanする。
2. nameまたは `FFF0` serviceで候補を選ぶ。
3. 接続する。
4. `FFF4` notifyを購読する。
5. `FFF1` へ初期化writeを送る。
6. 通知をparseし、切断またはtimeoutまで待つ。
7. 切断後はscanへ戻る。

notify購読後の推奨初期化:

| 接続後の目安 | コマンド     | バイト列                            |
| ------------ | ------------ | ----------------------------------- |
| 200 ms       | kg単位設定   | `fd 00 00 00 00 00 00 00 00 00 fd`  |
| 600 ms       | 現在時刻同期 | `f1 YYYY_HI YYYY_LO MM DD HH mm SS` |
| 1200 ms      | history要求  | `f2 00`                             |

単位設定コマンドの末尾は、それ以前のbyte列に対するXOR checksumです。
unit `0` の場合は末尾が `fd` になります。

時刻同期は現在ローカル時刻を次の形式で送ります。

| フィールド | サイズ  | エンコード          |
| ---------- | ------- | ------------------- |
| コマンド   | 1 byte  | `f1`                |
| 年         | 2 bytes | unsigned big-endian |
| 月         | 1 byte  | `1` to `12`         |
| 日         | 1 byte  | `1` to `31`         |
| 時         | 1 byte  | `0` to `23`         |
| 分         | 1 byte  | `0` to `59`         |
| 秒         | 1 byte  | `0` to `59`         |

例: `2026-06-20 23:51:20`

```text
f1 07 ea 06 14 17 33 14
```

## T9120系の通知

すべての通知は生バイト列を先に保存してください。既知の制御応答は次の通りです。

| バイト列 | 意味                          |
| -------- | ----------------------------- |
| `f1 00`  | 時刻同期成功候補              |
| `f2 00`  | history同期完了または受付候補 |
| `f2 01`  | history削除受付候補           |
| `f3 00`  | scale側の切断ack              |
| `cf ...` | 体重またはhistory payload     |

未知の通知でsessionを終了しないでください。生ログへ残して継続します。

### live weight packet

live weight packetは11 bytesで、先頭は `cf` です。

| byte範囲         | フィールド          | エンコード                     |
| ---------------- | ------------------- | ------------------------------ |
| `0`              | type                | `cf`                           |
| `1..2`           | impedance           | unsigned little-endian         |
| `3..4`           | weight raw          | unsigned little-endian         |
| `5..7`           | encrypted impedance | unsigned little-endian 24-bit  |
| `8` upper nibble | fat mode            | unsigned integer               |
| `9`              | status              | `0` 安定, `1` 動的, `2` 過荷重 |
| `10`             | checksum            | bytes `0..9` のXOR             |

kg単位の体重は `weight_raw / 100` です。

同じ測定値のstable packetが複数回来ることがあります。保存系を作る場合は
`weight_raw`、`impedance`、`encrypted_impedance` の組で最低500 ms程度
debounceしてください。ログ用途なら重複を見える形で出して構いません。

確認済みのstable packet例:

```text
cf e8 12 b4 14 b3 b6 9f 00 00 0f
```

parse結果:

| フィールド          | 値                       |
| ------------------- | ------------------------ |
| status              | stable                   |
| weight raw          | `5300`                   |
| weight              | `53.00 kg`               |
| impedance           | `4840`                   |
| encrypted impedance | `10466995`               |
| fat mode            | `0`                      |
| checksum            | XOR checksum `0f` と一致 |

### history packet候補

確認済み実行では履歴エントリは取得できていません。`cf` かつ18 bytesの通知が
来た場合は、live dataとは分けてhistory候補として記録します。

候補layout:

| byte範囲 | フィールド     | エンコード          |
| -------- | -------------- | ------------------- |
| `0..10`  | 測定フィールド | live packetと同じ   |
| `11..12` | 年             | unsigned big-endian |
| `13`     | 月             | `1` to `12`         |
| `14`     | 日             | `1` to `31`         |
| `15`     | 時             | `0` to `23`         |
| `16`     | 分             | `0` to `59`         |
| `17`     | 秒             | `0` to `59`         |

この経路を実機で確認するまでは、scale側history削除は送らないでください。

## T9140系のpacket候補

`T9140 V1/V2/V3` 系は別プロファイルです。生通知は `ac 02` から始まる候補を
優先して扱います。

既知の送信候補:

| コマンド         | バイト列                  |
| ---------------- | ------------------------- |
| 時刻取得         | `ac 02 f3 00 00 00 cc`    |
| 時刻同期template | `ac 02 f2 00 00 00 00 cc` |
| 単位取得         | `ac 02 fe 1b ff 00 cc`    |
| 単位設定template | `ac 02 fe 06 UNIT 00 cc`  |
| 測定完了ack      | `ac 02 fe 1d 00 00 cc`    |
| history要求      | `ac 02 ff 00 00 00 cf`    |

確認候補:

- 8 byte単位の通知が基本。
- 16または17 byte通知は、複数通知が結合している可能性がある。
- `byte[6] == ce` の候補は動的体重。
- `byte[6] == ca` の候補は安定体重。
- 体重raw候補は `byte[2] << 8 | byte[3]`。
- 19 byte通知は履歴エントリを2件含む可能性がある。

この系統はまだ実測していないため、最初の実装では生ログ優先にしてください。

## T9148/T9149/T9150/T9130系の扱い

これらは `FFF0` serviceを使う場合でも、`T9120`系の単純な `FFF4` notify +
`FFF1` writeだけでは不十分な可能性があります。

実装方針:

- `T9120`系パーサへ自動的に流さない。
- service/characteristic列挙結果を必ず出力する。
- primary notify、real-data notify、encrypt/write候補の生ログを分ける。
- 認証/暗号ハンドシェイクが必要そうな場合は、live weight仕様を別途定義する。
- 心拍、体脂肪開始、体重計側ユーザー選択らしきイベントは生ログに残す。

この系統をクリーンルームで扱う場合は、測定値仕様を確定する前に、
接続、全通知チャネルの購読、生通知の保存を確認事項として分けてください。

## 確認済み動作

`T9120`の実測では次の流れを確認しました。

1. `eufy T9120` と `FFF0` serviceを検出。
2. 接続し、`FFF4` notifyを購読。
3. 初期化writeを送信。
4. 次のstable packetを2回受信。

```text
cf e8 12 b4 14 b3 b6 9f 00 00 0f
```

5. `53.00 kg` としてparse。
6. その後 `f3 00` を受信して切断。

history要求は送信しましたが、この実行では履歴エントリは取得できていません。

## 未確認事項

- `T9120`が18 byte履歴エントリを返す条件。
- 未同期historyがある場合のwake状態や要求順序。
- encrypted impedanceから体組成をローカル計算できるか。
- kg以外のunit値がwire-level weight scalingへ影響するか。
- `T9140`系の実機packet。
- `T9148/T9149/T9150/T9130`系の認証/暗号handshakeと実測packet。
